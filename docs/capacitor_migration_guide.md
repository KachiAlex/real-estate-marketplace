# PropertyArk Capacitor Migration Guide

## Goal
Migrate the current React web app into a Capacitor-based Android app while keeping the existing backend, routing, and UI code as the source of truth.

## Why Capacitor
- The current custom WebView APK is fragile for popup-heavy flows.
- Capacitor provides a maintained mobile wrapper with better Android integration.
- It lets you keep the web app intact while packaging it for mobile.

## What stays the same
- React app in `src/`
- Backend APIs in `backend/`
- Production web build generated by `npm run build`
- Existing auth, payment, notification, and property flows

## What changes
- Add Capacitor to the web project
- Point Capacitor at the production build output
- Generate an Android project under a Capacitor-managed native wrapper
- Prefer native-safe fallbacks for any remaining popup-based browser behavior

## Recommended repo model
Use the existing repository as a single source of truth:
- `src/` for the web UI
- `backend/` for server APIs
- Capacitor Android project at the repo root or in a dedicated mobile folder

If you want to keep the custom WebView APK temporarily, treat it as a separate fallback path and do not build new features on it.

## Prerequisites
- Node.js and npm installed
- Android Studio installed
- Java/JDK configured for Android builds
- A successful web production build (`npm run build`)

## Step 1: Install Capacitor packages
Install these in the root web project:
- `@capacitor/core`
- `@capacitor/cli`
- `@capacitor/android`

If you later need iOS, add `@capacitor/ios` as well.

## Step 2: Initialize Capacitor
Create a Capacitor config using the built web app as the asset source.

Recommended values:
- App name: `PropertyArk`
- App ID: something like `com.propertyark.app`
- Web directory: `build`

## Step 3: Build and sync
Typical flow:
1. `npm run build`
2. `npx cap sync android`
3. Open the Android project in Android Studio
4. Run on an emulator or physical device

## Step 4: Configure Android permissions
At minimum:
- Internet access
- File/media permissions only if your file uploads/downloads require them

Review and keep permissions minimal.

## Step 5: Validate critical app flows
Test these on a real Android device:
- login / logout
- registration
- property browsing
- escrow/payment flows
- payment callback return flow
- receipt viewing/downloading
- inspection scheduling
- notifications

## Step 6: Decide what to do with browser-only features
Keep or replace based on need:
- Popup payment fallback: keep only if it still works well on device
- Browser print preview: consider in-app PDF/download later
- Google Calendar shortcuts: already removed from the active UI path
- Legacy Google auth callback: already neutralized

## Step 7: Optional improvements after the first build
Once the basic APK works:
- Add Capacitor plugins for native dialogs, storage, and deep linking if needed
- Replace popup-based payment flows with more mobile-friendly UX
- Add native splash screen and icon polish
- Consider push notification integration if required

## Suggested implementation order
1. Add Capacitor packages to the root project
2. Initialize Capacitor with `build` as the web directory
3. Generate Android project and open it in Android Studio
4. Run a production build and sync assets
5. Test login, property browsing, and payment flows
6. Only then decide whether to refactor any remaining popup-heavy screens

## Notes for this repo
- The custom Android WebView wrapper under `mobile/WebViewAPK` is separate from Capacitor and should not be expanded further if you choose Capacitor.
- The React app is already cleaned up from Google OAuth and Google Calendar popup shortcuts, which reduces mobile risk.
- Existing popup guards should help during migration, but Capacitor is still the better long-term shell.

## Next action recommendation
If you want to proceed, the next practical step is:
1. install Capacitor packages
2. create the Capacitor config
3. sync the Android project
4. test the production build on device

## Rollback strategy
If Capacitor introduces issues:
- keep the web app unchanged
- continue shipping the existing web build
- use the current Android WebView APK only as a temporary fallback
